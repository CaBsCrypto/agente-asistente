What it is

A trusted layer between people, agents, apps and payments.

agent-assistant pairs a simple chat for end users with MCP, WebMCP and API surfaces for developers. Every sensitive action is constrained: identity is separated from consent, consent from payment; intents are frozen and hashed before approval; execution runs exactly once and leaves durable evidence. The same engine lets other agents use us, and lets us use other apps.

MVP BOUNDARY

Honest status. Authentication, Stellar wallet creation, market data, persistent user state, the replay-safe commerce demo and a real Testnet x402 payment are live. Notion and DeFindex Testnet signing are implemented and awaiting complete user acceptance. Merchant fulfillment and mainnet settlement are not live. Research, sandbox behavior and production integrations are never conflated.

Architecture

Layered, non-custodial, evidence-first.

A Next.js app and chat front the system. Privy owns identity and the wallet; the private key never reaches our server. A deterministic router dispatches to connectors and to a reusable LangGraph safe-action engine. State, intents, workflows and receipts persist in Neon Postgres.

Browser

Consent & signature

Privy session and the visible confirmation live here. A transaction-specific signature is produced in the browser and never leaves it as a raw key.

Server

Route · verify · record

Verifies identity, routes tools, evaluates policy before any execution, encrypts OAuth tokens, and writes audit records and receipts.

Providers

MCP · APIs · network

External MCP servers, the Privy wallet service and the Stellar network — reached through bounded connectors with verified evidence.

The safe-action lifecycle

One universal path for any risky action.

Every write or payment runs through a reusable LangGraph StateGraph of seven bounded nodes. A connector plugs in three methods — prepare, execute, verify — and inherits approval, idempotency and evidence for free.

01validateResolve connector + capability; check the operation matches.
02connectionIs the provider linked? If not, pause for connection.
03prepareBuild an immutable action; hash it into a frozen digest.
04policyFinancial or write → approval required; reads may pass.
05approveResume only if approver + digest exactly match.
06execute onceworkflowId is the idempotency key; never twice.
07verifyComplete only when the connector returns verified evidence.
Frozen intent

sha256 binding

The prepared action is canonicalized (sorted keys) and SHA-256 hashed. Change any parameter and the digest changes — invalidating a prior approval.

Replay safety

Execute once

The workflow id is the idempotency key. Inserts use conflict-do-nothing, completed workflows are immutable, and a duplicate confirm returns the same result.

Durable trail

Every step logged

Each checkpoint appends a versioned event. Optimistic concurrency guards against races; the whole path is auditable in Neon.

Bidirectional MCP gateway

Other agents use us. We use other apps.

The Model Context Protocol runs in both directions. Inbound, three servers let external agents and providers discover offers, freeze intents and (in sandbox) execute. Outbound, we act as an MCP/OAuth client into Notion, Travala and market data — all governed by the same policy and evidence layer.

Security model

Eight principles, enforced in code.

Not slogans — each maps to a concrete guard in the codebase.

Non-custodial

The orchestration layer never holds a private key. Signing happens in the browser via Privy; the server only verifies the Ed25519 signature. The wallet is owned by the user.

Least privilege

OAuth requests the minimum scopes; MCP tokens carry an explicit scope set enforced per subject; provider keys are scoped to one catalog.

Explicit authority

Identity, connection consent and payment approval are three separate acts. The approval gate requires an exact userId + digest match; payments require explicitConfirmation: true.

Frozen intent

Merchant, amount, network and expiry are pinned into an immutable action and SHA-256 hashed before approval. The signing hash is re-checked at signing time.

Replay safety

Idempotency keys, unique DB indexes and status guards mean a duplicate request returns the previous result — never a second debit.

Encrypted credentials

External OAuth tokens, PKCE verifiers and client secrets are encrypted at rest with AES-256-GCM and never returned to the browser.

Independent evidence

A connector must return verified evidence to complete. Settlement (a transaction hash) and fulfillment (delivery) are separate states.

Honest failure

Ambiguous external results are not blindly retried. A timeout or unclear response sets a reconciliation state and stops automatic mutation.
